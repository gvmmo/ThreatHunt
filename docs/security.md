# Security Model — ThreatHunt

ThreatHunt is a platform that runs OSINT tooling against real targets. Its own security is therefore critical — a compromised instance could be misused to gather intelligence against unauthorized targets. Security is implemented across infrastructure, application and operational layers.

---

## Layers

### 1. Perimeter — AWS WAF

Web Application Firewall configured with four managed rule groups:

| Rule group | Purpose |
|------------|---------|
| Anonymous IP list | Blocks traffic from VPNs, Tor exit nodes and anonymizing proxies |
| Core rule set (CRS) | OWASP Top 10 protection — SQLi, XSS, path traversal and more |
| Known bad inputs | Blocks requests matching signatures of known exploit frameworks |
| Amazon IP reputation list | Blocks IPs flagged by AWS threat intelligence |

Priorities are set to minimize false positives while maximizing coverage against automated attack tools.

### 2. Network segmentation

| Tier | Internet access | Reachable from |
|------|----------------|----------------|
| Web instances (public subnet) | Yes | Internet via ALB |
| MongoDB nodes (private subnet) | No | Web instances only (port 27017) |
| S3 bucket | Via AWS endpoints | Web instances only |

Security groups enforce default-deny at the network level. MongoDB ports are never exposed to the internet.

### 3. Infrastructure access

- All SSH access restricted to specific IP ranges
- Password authentication disabled on all instances
- MongoDB Replica Set uses keyfile authentication between nodes
- S3 bucket access restricted to application IAM role only

### 4. Application security

**Authentication**
- User registration and login system with hashed passwords
- Session management restricts dashboard access to authenticated users only
- The main platform route is authentication-gated — no unauthenticated access to scan functionality

**Input validation**
- All scan targets validated before reaching the orchestrator
- Duplicate scan detection prevents redundant resource consumption
- API keys stored per user, never shared between accounts or exposed in logs or reports

**Data isolation**
- Each user's scan history is scoped to their account
- No cross-user data access is possible through the API

### 5. Logging and monitoring

- AWS CloudWatch monitors CPU usage and ALB latency with configured alarms
- Per-scan execution logs generated by Python's `logging` module with timestamps
- Web server access and error logs maintained separately
- Log files stored in `logs/` with rotation support

---

## Responsible use

ThreatHunt is designed for authorized security assessments only. Users are responsible for ensuring they have explicit permission to scan any target they submit to the platform.

**Passive scans** retrieve publicly available information and do not interact directly with the target system. They are suitable for initial reconnaissance and do not generate alerts on the target side.

**Active scans** interact directly with the target and may trigger IDS/IPS alerts. They should only be used against systems the user owns or has written authorization to test.

Misuse of the platform against unauthorized targets is a violation of the terms of use and may constitute a criminal offense under applicable law.

---

## Security recommendations for administrators

- Keep all Python dependencies updated (`pip install -r requirements.txt --upgrade`)
- Rotate API keys regularly from the Profile section
- Monitor `logs/` for unusual scan activity or authentication errors
- Restrict MongoDB access to the application host only
- Use strong, unique passwords for all user accounts
- Back up the MongoDB database regularly using `mongodump`

---

## Reporting vulnerabilities

If you discover a security vulnerability in ThreatHunt, please report it responsibly by contacting the development team directly rather than disclosing it publicly.
